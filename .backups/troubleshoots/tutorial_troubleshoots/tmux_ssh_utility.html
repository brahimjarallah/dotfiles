Great question — Kitty vs tmux, especially in a Hyprland + Arch context, is nuanced. Let’s break it down as an expert would:

✅ Why not tmux (or: when not to use it)?

tmux is legendary on X11 or TTY setups, but under Wayland, especially with Hyprland, it has limitations:

🧱 Tmux is a TTY multiplexer, not a Wayland-native tool:

  • Runs inside a terminal — it doesn’t know about GUI layout, window management, or HiDPI.

  • Doesn’t support ligatures, images, emoji well unless the terminal supports them (Kitty, WezTerm, etc.).

  • Copy-paste is a mess under Wayland unless you integrate with tools like wl-clipboard.

❌ Redundant with modern terminals:

  • Kitty, WezTerm, etc., can natively do tabs, splits, session restore — no need for tmux.

  • These terminals offer better font rendering, GPU acceleration, and clipboard integration.

🧩 Input issues:

  • Tmux can have keybinding weirdness or latency over ssh under Wayland.

  • Multiplexing in GUI terminals often behaves better than tmux in a nested shell.

✅ When to use tmux (still valid cases):

  • Remote SSH sessions (tmux shines here).

  • Need persistent sessions across reboots (e.g., for dev servers).

  • TUI workflows (e.g., htop, vim, ncdu) that you want session-managed.

⚔️ Kitty vs Tmux — which to choose?

🐱 Kitty (with native features):

  • GPU-accelerated, fast, crisp.

  • Has layouts, tabs, splits, even broadcast input (kitten @ send-text).

  • Wayland-comfy, despite being XWayland.

  • Kitty’s scripting (kitty +kitten) makes it tmux-like without the mess.

🔥 If you’re on Hyprland, Kitty’s built-in features replace 90% of tmux needs.

🧮 Combined? Kitty + tmux?

You can still use tmux inside kitty, but it’s usually redundant:

  • Nested keybindings (e.g., ctrl+b vs kitty’s own keys).

  • Clipboard sync issues.

  • Less pretty output (no GPU).

Great — you're asking the right questions for a serious Hyprland + SSH + tmux setup. Let’s walk through how to use tmux for remote SSH sessions and make them persistent across reboots, like a pro.

✅ Why Use tmux Over SSH?

When you SSH into a remote server:

  • If your connection drops or you reboot your local system, the session is lost… unless you use tmux.

  • tmux detaches your terminal from your SSH session and keeps it alive on the server.

🔧 Step-by-Step: Use tmux for Remote SSH + Persistent Sessions

1. Install tmux on the remote server

bash
sudo pacman -S tmux         # Arch-based
# or for Debian/Ubuntu:
sudo apt install tmux

2. Start or attach to a session

bash
tmux new -s dev

  • This creates a session named dev.

  • Inside this, you can run long-lived apps: editors, servers, builds, etc.

💡 Tip: You can have multiple sessions — e.g., tmux new -s staging, tmux new -s monitor

3. Detach from session without stopping it

Press:

bash
Ctrl + b, then d

This detaches the session, but all your processes keep running.

4. Re-attach to a tmux session

After reconnecting via SSH:

bash
tmux attach -t dev
# or list all:
tmux ls

5. Auto-start tmux on SSH login (optional, advanced)

In your ~/.bashrc or ~/.zshrc on the remote server, add:

bash
# auto-start tmux if not already inside one
[ -z "$TMUX" ] && tmux attach -t default || tmux new -s default

This means: every time you SSH, you’ll resume your tmux session automatically.

6. Persist sessions across remote reboots

Tmux itself won't survive reboots, but here’s how to keep dev environments always alive:

✅ Option A: Use tmux + systemd user service (recommended)

  1. Create a systemd unit:
    ~/.config/systemd/user/tmux@dev.service

ini
[Unit]
Description=tmux session dev
After=network.target

[Service]
Type=oneshot
ExecStart=/usr/bin/tmux new-session -d -s dev
RemainAfterExit=yes

[Install]
WantedBy=default.target

  1. Enable at boot:

bash
systemctl --user daemon-reexec
systemctl --user enable tmux@dev.service

✅ Now when the server boots and you SSH in, your tmux session dev will already be running.

🔄 Combine with SSH config

You can also make SSH smoother from your local Hyprland machine:

ini
# ~/.ssh/config
Host dev-server
    HostName 192.168.0.123
    User myuser
    IdentityFile ~/.ssh/id_ed25519

Then just:

bash
ssh dev-server
